# פרויקט מתמשך אחד — "AtlasDesk" (במקום פרויקטים מנותקים)

## החלטה (2026-07-12, בעקבות בקשת המשתמש)
במקום שכל מודול AI ייגמר בפרויקט קטן ומנותק (כמו שהיה עד כה: מסווג ספאם, מזהה ספרות, מחשבון
טוקנים — כולם עומדים בפני עצמם), **מהמודול הבא ואילך** כל מודול AI מוסיף יכולת אחת אמיתית לאותה
אפליקציה מסחרית אחת שגדלה לאורך כל האקדמיה. בסיום האקדמיה לסטודנט תהיה פלטפורמת AI production-grade
מלאה שהוא בנה בעצמו, יכולת אחר יכולת.

**הפרויקטים הקיימים (spam classifier, digit recognizer, token calculator וכו') נשארים כפי שהם** —
הם היו חלק מטראק "יסודות AI" הבסיסי ומטרתם להמחיש מושגים בודדים. השינוי חל על הטראקים הבאים
(Prompt Engineering, MCP/RAG/Agents integration, Production) שהם הליבה האמיתית של מטרת המשתמש.

## מהי AtlasDesk?
אפליקציית תמיכת לקוחות AI מסחרית (AI Customer Support Copilot) — נבחרה כי היא ממחישה טבעית כל
יכולת מהרשימה של המשתמש: היא צריכה שיחה עם לקוחות (LLM+prompting), ידע מוצר (RAG+embeddings),
פעולות אמיתיות (tool calling — בדיקת סטטוס הזמנה, פתיחת טיקט), אסקלציה בין סוכנים (multi-agent),
ולוח בקרה לצוות תמיכה (dashboard+auth+monitoring+billing). זה גם תרחיש עסקי אמיתי לגמרי, לא תרגיל מלאכותי.

## מבנה טכני
- **מיקום בקוד:** `app/src/app/(atlasdesk)/` — route group נפרד מהאקדמיה עצמה, כדי שהאפליקציה הנבנית
  תהיה מוצר עצמאי אמיתי (לא "בתוך" ממשק הלימוד), אבל באותו repo/deployment.
  - הסטודנט תמיד יכול לפתוח את `/app` (או subdomain בעתיד) ולראות את המוצר שהוא בונה.
- **כל שיעור פרויקט** מקשר לקוד המדויק שנוסף (קובץ/רכיב חדש) ומסביר אותו לפי אותו קובץ — לא דוגמת
  קוד מנותקת.
- **מעקב יכולות:** קובץ `docs/13-atlasdesk-features.md` (ראה למטה) מתעד אילו יכולות כבר נבנו,
  בהתאמה לרשימת היכולות שהמשתמש ביקש.

## מיפוי טראקים ← יכולות AtlasDesk (תוכנית)
| טראק/מודול | יכולת שנוספת ל-AtlasDesk |
|---|---|
| Prompt Engineering | מנוע השיחה הבסיסי: system prompt מובנה, few-shot examples, streaming responses |
| Claude Code ופיתוח AI-assisted | תיעוד/הדגמת workflow פיתוח על הקוד של AtlasDesk עצמו |
| MCP ו-Tool Calling | כלי אמיתי: "בדוק סטטוס הזמנה", "צור טיקט" — קריאות פונקציה אמיתיות מה-LLM |
| Embeddings ו-Vector DB | בסיס ידע מוצר מוטמע (embeddings) לחיפוש סמנטי |
| RAG | תשובות מבוססות מסמכי מוצר אמיתיים, לא רק ידע כללי של המודל |
| Fine-tuning concepts | הערכת ביצועי AtlasDesk, A/B בין prompts |
| סוכן AI בודד | הסוכן שמטפל בשיחה קצה-לקצה עם זיכרון שיחה |
| Multi-Agent | אסקלציה: סוכן-תמיכה-ראשוני → סוכן-מומחה-טכני → סוכן-סיכום |
| אוטומציה/Scraping/APIs | אינטגרציה עם API חיצוני מדומה (מערכת הזמנות) |
| ניטור וסקייל | דשבורד ניטור שיחות, latency, שגיאות |
| עלויות ואבטחת AI | מעקב עלויות טוקנים לכל שיחה, הגנת prompt injection |
| Best Practices לפרודקשן | feature flags, rollback, health checks |
| SaaS/מסחור | Auth, ניהול משתמשים, billing simulation, admin panel, פריסה אמיתית |

## עקרון מנחה
כל שיעור פרויקט מ-AtlasDesk חייב:
1. להוסיף קוד עובד אמיתי לאפליקציה (לא placeholder / "בעתיד נוסיף")
2. להיבנות על מה שנבנה בשיעורים הקודמים — לא "מתחילים מחדש"
3. לכלול את סעיף EngineeringInsights (למה/חלופות/מתי-לא/עלות/אבטחה/סקייל/תחזוקה/מערכות-אמיתיות)
4. לעבור build+typecheck+בדיקה ידנית לפני סיום המודול, בדיוק כמו כל שיעור אחר
